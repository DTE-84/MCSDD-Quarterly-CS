01 // The Challenge
Marion County MCSDD support coordinators were completing quarterly reviews by hand — with no standardized structure, no enforcement of DMH-required clinical language, and no systematic tie-back to the Person-Centered Support Plan. At peak caseload, compliance wasn't a process. It was a liability.

The Missouri DMH TCM Technical Assistance Manual requires quarterly reviews to evaluate six distinct domains: documents reviewed, service delivery against the PCSP, outcomes progress, health and safety, rights and dignity, and incident reporting. Done manually, a single review could consume 30–45 minutes — and still return from audit flagged for missing active phrasing, undocumented service delivery status, or absent rights observations.

The downstream cost wasn't just time. Inconsistent quarterly documentation created compounding audit exposure across the entire caseload. Coordinators were producing reviews that were technically complete but clinically uneven — some thorough, some thin, none standardized. When auditors reviewed a file, the quality of documentation depended entirely on who wrote it and how much time they had that day.

The second problem was architectural. The quarterly review exists to evaluate the PCSP — yet the two documents lived in completely separate workflows. Coordinators had to manually cross-reference the plan while completing the review. There was no mechanism to enforce that every active PCSP outcome had a corresponding quarterly progress entry. The link between the plan and its review was entirely dependent on individual memory.

02 // The Solution

Quarterly Review Pro is a zero-footprint, HIPAA-compliant documentation engine built as a direct companion to PCSP Assistant Pro. It enforces the full DMH TCM six-domain quarterly review structure through a guided form interface, generates compliant case note or summary report narratives via AI, and outputs print-ready documentation — all from a single HTML file that runs entirely in the browser with no installation, no server, and no residual data.

⬡ PCSP-Linked Review Framework
Quarterly outcomes and service monitoring sections map directly to the PCSP structure — the same domain architecture from PCSP Assistant Pro. Every outcome entered in the quarterly review corresponds to a PCSP goal area, maintaining the clinical chain of evidence auditors require from plan to quarterly to case note.
⬡ DMH TCM Six-Domain Structure
Built against the Missouri DMH Targeted Case Management Technical Assistance Manual. Sections enforce: contact type and face-to-face tracking (one required annually), documents reviewed checklist, service delivery monitoring per PCSP, outcomes progress with status classification, health and safety with rights and dignity observation, and CIMOR-referenced incident reporting.
⬡ Dual Narrative Output — Nova Engine
The integrated Nova AI engine generates two distinct output formats from the same structured input: a Case Note beginning with the required "Quarterly Review" prefix per DMH documentation standards, and a formal Summary Report formatted for individual file retention. Clinical language, active phrasing, and domain coverage are enforced at the prompt level — not revised after the fact.
⬡ Service Monitoring Repeater
An unbounded dynamic service roster allows coordinators to document each active PCSP service independently — delivery status against plan, adequacy assessment, provider, and monitoring observations. Each entry serializes cleanly for AI narrative generation, ensuring every service in the PCSP has a corresponding quarterly monitoring record.
⬡ Outcomes Progress Engine
PCSP outcomes are entered as dynamic repeating entries with six status classifications — Making Progress, Maintaining Level, Achieved, Regression, Insufficient Data, Not Initiated — plus priority weighting and evidence notes. The status vocabulary maps directly to the language DMH auditors expect to find in quarterly documentation.
⬡ Contact Type Compliance Tracker
Explicitly captures whether the quarterly contact was face-to-face or phone, with a date field for face-to-face documentation. The DMH TCM manual requires at minimum one face-to-face contact annually — this field enforces awareness of that requirement and surfaces it in the generated narrative.

05 // Technical Appendix
I. Data Lifecycle Management

The primary security feature is its Stateless Architecture. Unlike SaaS platforms, this tool functions as a pure client-side processor — data exists only for the duration of the session.

Zero-Persistence
All logic executes in browser RAM. Sensitive PII exists only in volatile memory during a live session. Tab close is a full data wipe — no residual trace in storage, network logs, or cache.
The "Refresh" Wipe
Closing or refreshing the tab instantly purges all session data. No database, no POST requests, no residual trace. The next session begins as a clean state.
Nova API Privacy Boundary
AI narrative generation transmits only anonymized clinical field values — service statuses, outcome descriptions, domain checkboxes. Direct identifiers (names, SSNs, DOBs) are never included in the API payload. The HIPAA notice is displayed persistently in the interface.
II. DMH TCM Six-Domain Structure

The review structure derives directly from the Missouri DMH Targeted Case Management Technical Assistance Manual's definition of quarterly case monitoring. Each domain maps to a dedicated form section.

  • 01 Contact Type & Documents Reviewed — face-to-face tracking, document checklist
  • 02 Service Monitoring — delivery vs PCSP, adequacy assessment, provider, notes
  • 03 PCSP Outcomes Progress — status classification, priority, evidence
  • 04 Health, Safety & Rights — health events, safety flags, rights observation, community
  • 05 Incidents & Crisis — incident log, CIMOR reference, reporting status
  • 06 Conclusions & Recommendations — plan revision decision, action items, next review
III. Dynamic Data Architecture

Services and outcomes are managed as in-memory JavaScript arrays, rendered to the DOM on every state change and serialized into the Nova prompt with full fidelity.

servicesData = [] // name → delivered → adequate → provider → notes outcomesData = [] // name → status → priority → evidence actionsData = [] // action → owner → due date all arrays serialize into Nova prompt payload on generate no arrays persist beyond the active browser session
IV. Nova Prompt Architecture — Dual Mode

Two distinct system prompts govern output format. Mode selection is explicit — the coordinator chooses the output type before generating.

Case Note Mode: prefix → "Quarterly Review" (DMH required) voice → third-person past tense length → concise — no duplicate of underlying logs domains → all six, condensed into single note Summary Report Mode: format → formal section headers per TCM manual retention → filed with individual record or under own heading language → active voice, clinical — audit-ready on first draft domains → all six, each as a distinct section

Note: Both modes instruct Nova to reference "the individual" — never full names or direct identifiers. PHI sanitization is enforced at the prompt level, not as a post-processing step.

The quarterly review tool and PCSP tool occupy the same directory, functioning as a unified documentation suite within a single file system location.
